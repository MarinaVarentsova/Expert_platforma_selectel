БАГ НЕ ИСПРАВЛЕН.

Конкретный сценарий, который сейчас ломается:

1. Заказчик открыл карточку заказа.
2. В карточке видит двух предложенных экспертов.
3. Заказчик нажал "Выбрать эксперта" по эксперту 1.
4. Эксперт 1 взял заказ в работу.
5. Заказчик НЕ обновляет страницу.
6. В этой же старой карточке заказа заказчик нажимает "Выбрать эксперта" по эксперту 2.
7. Сейчас кнопка отрабатывает и эксперту 2 уходит уведомление.

Так быть НЕ должно.

Требование:

Именно в обработчике кнопки "Выбрать эксперта" В КАРТОЧКЕ ЗАКАЗА, прямо перед отправкой уведомления эксперту и перед любым update, добавить свежую проверку БД.

Не при загрузке страницы.
Не в useEffect.
Не только в списке.
А именно ВНУТРИ onClick/handler кнопки "Выбрать эксперта".

Проверка:

Перед отправкой уведомления эксперту 2 выполнить fresh select:

select *
from palata_request_matches
where request_id = currentRequestId
and status = '<фактический статус в работе>';

или использовать фактическое поле/статус проекта, которым отмечается эксперт, взявший заказ в работу.

Если найден хотя бы один эксперт в работе:

1. НЕ отправлять уведомление эксперту 2.
2. НЕ обновлять palata_request_matches по эксперту 2.
3. НЕ обновлять palata_request_contacts по эксперту 2.
4. НЕ менять статус заказа.
5. Показать заказчику сообщение:

"На заказ уже назначен эксперт"

6. Обновить данные карточки заказа.
7. Скрыть/заблокировать кнопки выбора других экспертов.

Если НЕ найден эксперт в работе:

* выполнять текущую логику выбора эксперта.

Важно:
Проверка должна быть синхронно перед notification.

Правильный порядок:

handleChooseExpert(expert2) {
const assigned = await checkRequestAlreadyAssigned(requestId)

if (assigned) {
showMessage("На заказ уже назначен эксперт")
await refreshRequest()
return
}

// только здесь можно отправлять уведомление эксперту 2
await notifyExpert(...)
await updateMatch(...)
}

Запрещено:

* отправлять уведомление до проверки;
* делать проверку только при рендере;
* полагаться на старый state страницы;
* чинить только UI без fresh select из БД.

Acceptance test:

1. Открыть заказчиком карточку заказа с 2 экспертами.
2. Выбрать эксперта 1.
3. Под экспертом 1 взять заказ в работу.
4. Не обновляя страницу заказчика, нажать "Выбрать эксперта" по эксперту 2.
5. Ожидаем:

   * появляется сообщение "На заказ уже назначен эксперт";
   * в Network нет вызова уведомления эксперту 2;
   * статус эксперта 2 не меняется;
   * данные карточки обновляются.

Вернуть:

1. Точный файл и функцию обработчика кнопки "Выбрать эксперта" в карточке заказа.
2. Код функции checkRequestAlreadyAssigned.
3. Лог в консоли:
   [choose-expert] fresh assigned check
   [choose-expert] blocked because request already assigned
4. Подтверждение, что notification вызывается только после проверки.
