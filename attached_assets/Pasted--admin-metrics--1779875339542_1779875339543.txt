Нужно переделать страницу `/admin/metrics`.

Сейчас метрики считаются несогласованно: количество заказов в одном блоке не совпадает с количеством заказов в другом блоке.
Сначала нужно описать и вернуть в чат формулы расчёта каждого показателя, потом привести UI и расчёты к единой логике.

## 1. Сначала вернуть формулы

Перед изменением интерфейса напиши в чате формулы для каждого показателя:

* всего заявок;
* выполнено заявок;
* % выполненных заявок;
* среднее время выполнения заказа в днях;
* средний рейтинг заказчика;
* средний рейтинг эксперта;
* всего заказчиков;
* активных заказчиков;
* % активных заказчиков;
* всего экспертов;
* активных экспертов;
* % активных экспертов;
* эксперты в Реестре Палаты СЭ;
* эксперты в Центр судэксперт;
* новые заявки;
* подбор эксперта;
* выбор эксперта;
* в работе;
* выполнено;
* неактуальные;
* не нашли эксперта;
* отказ эксперта;
* распределение заказов по регионам;
* распределение заказов по направлениям;
* распределение экспертов по регионам;
* распределение экспертов по направлениям.

Важно:
для каждого показателя указать:

* таблицу-источник;
* поле;
* фильтр;
* формулу;
* какие статусы учитываются.

## 2. Новая структура страницы метрик

Разделить страницу на зоны.

### Зона 1. Верхняя строка

Верхняя строка должна содержать:

1. Всего заявок
2. Выполнено заявок
3. % выполненных заявок
4. Среднее время выполнения заказа в днях

Формула:

* всего заявок = count из `palata_requests`
* выполнено = count, где status = `completed`
* % выполненных = completed / total * 100
* среднее время выполнения = среднее значение между `created_at` и датой перевода в `completed`

Если события завершения есть в `palata_status_events`, лучше брать дату completion оттуда. Если нет — временно использовать `palata_requests.updated_at`.

### Зона 2. Правая панель

Сделать правую вертикальную панель со справочными метриками:

1. Средний рейтинг заказчика
2. Средний рейтинг эксперта
3. Всего заказчиков
4. Активных заказчиков + %
5. Всего экспертов
6. Активных экспертов + %
7. Реестр Палаты СЭ
8. Центр судэксперт

Формулы:

* средний рейтинг заказчика = avg из `palata_customer_ratings.rating`
* средний рейтинг эксперта = avg из `palata_expert_ratings.rating`
* всего заказчиков = count из `palata_customer_profiles`
* активных заказчиков = count distinct заказчиков, у которых есть заявки со status != `cancelled` / `not_actual` / `irrelevant`
* % активных заказчиков = active_customers / total_customers * 100
* всего экспертов = count из `palata_expert_profiles`
* активных экспертов = count distinct экспертов, у которых есть заявки в статусе `in_work`
* % активных экспертов = active_experts / total_experts * 100
* Реестр Палаты СЭ = count экспертов, где `palata_registry_verified = true`
* Центр судэксперт = count экспертов, где `centrsudexpert_verified = true`

Если в enum статус не `not_actual`, а другой — использовать фактическое значение из схемы.

### Зона 3. Блок «Заявки»

Под верхней строкой сделать блок «Заявки».

#### Дорожка 1. Основная воронка

На каждом кубике показывать:

* количество;
* процент от всего количества заявок.

Последовательность:

1. Всего
2. Новые
3. Подбор эксперта
4. Выбор эксперта
5. В работе
6. Выполнено
7. Неактуальные

Это не conversion funnel, а раскладка текущего состояния заявок по статусам.

Формулы:

* Всего = count всех заявок
* Новые = count status = `new`
* Подбор эксперта = count status = `matching`
* Выбор эксперта = count status = `expert_selection`
* В работе = count status = `in_work`
* Выполнено = count status = `completed`
* Неактуальные = count status = `not_actual` / `irrelevant` / фактический enum

Процент каждого кубика = count_status / total_requests * 100.

#### Дорожка 2. Проблемная воронка

На каждом кубике показывать:

* количество;
* процент от всего количества заявок.

Последовательность:

1. Всего заявок
2. Не нашли эксперта
3. Отказ эксперта
4. Неактуальные

Формулы:

* Всего заявок = count всех заявок
* Не нашли эксперта = count заявок, где status = `matching` и нет активных matches, либо есть event `no_experts_found`
* Отказ эксперта = count заявок, где по всем предложенным экспертам status = `declined`, либо есть declined в `palata_request_matches`
* Неактуальные = count status = `not_actual` / `irrelevant`

Процент каждого кубика = count / total_requests * 100.

## 4. Распределения

Ниже добавить блок «Распределение».

Нужно сделать переключатель:

* Заказы
* Эксперты

И внутри каждого режима две вкладки / два блока:

* По регионам
* По направлениям

### Если выбран режим «Заказы»

Показывать:

* распределение заказов по регионам из `palata_requests.region`;
* распределение заказов по направлениям из `palata_requests.specialization` / фактического поля направления.

### Если выбран режим «Эксперты»

Показывать:

* распределение экспертов по регионам из `palata_expert_profiles.regions`;
* распределение экспертов по направлениям из `palata_expert_profiles.specializations`.

Для массивов использовать unnest / обработку массива на frontend.

## 5. Важное требование

Все метрики должны считаться из одного общего источника данных на странице.

Нельзя, чтобы:

* верхние карточки считались одним запросом;
* воронка другим запросом с другими фильтрами;
* распределение третьим запросом с другой логикой.

Нужно сделать единый metrics service / helper:

* получает данные из Supabase;
* нормализует статусы;
* считает все показатели;
* UI только отображает результат.
