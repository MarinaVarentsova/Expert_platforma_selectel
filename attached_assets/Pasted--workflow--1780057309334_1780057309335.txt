# ЭТАЛОННАЯ ЛОГИКА ПРОДУКТА

В проекте есть две разные зоны работы с заказами:

## 1. Канбан

Канбан остаётся в ЛК заказчика, ЛК эксперта и админке.

Назначение канбана:

* общий обзор заказов;
* распределение по статусам;
* контроль pipeline.

Канбан НЕ должен заменяться Action Center.

## 2. Action Center / «Требуют действия»

Action Center — это отдельная зона, где пользователь видит только те задачи, по которым сейчас требуется его действие.

Примеры:

* заказчику нужно выбрать эксперта;
* эксперту нужно ответить на назначение;
* заказчику нужно согласовать дату;
* эксперту нужно подтвердить финальное назначение;
* заказчику нужно оценить эксперта;
* эксперту нужно оценить заказчика.

# СЦЕНАРИЙ 1. Создание заказа заказчиком

Заказчик создаёт заказ в ЛК заказчика.

Ожидаемое поведение:

1. Создаётся запись в `palata_requests`.
2. Статус заказа = `new`.
3. Создаётся событие в `palata_status_events`.
4. Автоматически запускается подбор экспертов.

Проверить:

* есть ли форма создания заказа;
* сохраняются ли данные в Supabase;
* создаётся ли событие;
* запускается ли matching;
* какие поля реально используются.

---

# СЦЕНАРИЙ 2. Автоматический подбор экспертов

Система ищет экспертов по:

1. направлению экспертизы;
2. региону;
3. готовности к командировкам;
4. активности профиля;
5. доступности;
6. рейтингу;
7. регистрации в Палата судебных экспертов РФ;
8. регистрации в Центр судэксперт;
9. исключению экспертов, которые уже отказались по этому заказу.

Если эксперты найдены:

1. Создаются записи в `palata_request_matches`.
2. Заказ получает статус `expert_selection`.
3. Заказчику создаётся action item:
   `experts_matched`.

Если эксперты не найдены:

1. Заказ остаётся / переходит в `matching`.
2. Создаётся событие:
   `no_experts_found`.
3. Админу создаётся action item:
   `manual_matching_required`.

Проверить:

* есть ли matching engine;
* где он вызывается;
* как считается score;
* создаются ли matches;
* создаётся ли action item заказчику;
* исключаются ли отказавшиеся эксперты.

# СЦЕНАРИЙ 3. Заказчик видит подобранных экспертов

В ЛК заказчика во вкладке «Требуют действия» появляется action item:

`experts_matched`

Заказчик открывает заказ и видит список подобранных экспертов.

В карточке эксперта должны быть:

* ФИО;
* рейтинг;
* регионы работы;
* направления экспертиз;
* готовность к командировкам;
* регистрация в Палата судебных экспертов РФ;
* номер регистрации в Палата судебных экспертов РФ;
* регистрация в Центр судэксперт;
* номер регистрации в Центр судэксперт;
* количество выполненных заказов;
* описание опыта.

Проверить:

* есть ли вкладка «Требуют действия» у заказчика;
* есть ли badge-счётчик;
* показываются ли подобранные эксперты;
* все ли поля карточки эксперта выводятся;
* открывается ли карточка заказа.

---

# СЦЕНАРИЙ 4. Заказчик выбирает эксперта

Заказчик выбирает одного эксперта из подобранных.

После выбора:

1. Выбранный match получает статус:
   `selected_by_customer`.
2. Создаётся / обновляется запись в `palata_request_contacts`.
3. Заказчику открываются контакты эксперта.
4. Эксперту открываются контакты заказчика.
5. Action item заказчика `experts_matched` закрывается.
6. Эксперту создаётся action item:
   `customer_selected_you`.
7. Создаётся событие в `palata_status_events`.
8. Создаётся test email event в `palata_email_events`.

Проверить:

* реализована ли кнопка выбора эксперта;
* меняется ли статус match;
* открываются ли контакты;
* создаётся ли action item эксперту;
* закрывается ли action item заказчика;
* пишутся ли события и email events.

# СЦЕНАРИЙ 5. Эксперт отвечает на выбор заказчика

Эксперт во вкладке «Требуют действия» видит action item:

`customer_selected_you`

Эксперт может выбрать:

1. «Взять в работу»
2. «Могу начать с даты»
3. «Отказаться»

---

## 5.1. Эксперт нажал «Взять в работу»

Ожидаемое поведение:

1. `palata_requests.status = in_work`.
2. Текущий match = `accepted_work`.
3. Остальные matches по заказу = `closed_by_other_expert`.
4. Action item эксперта закрывается.
5. Открытые action items других экспертов по этому заказу отменяются.
6. Заказчику создаётся action item:
   `expert_started_work`.
7. Создаётся событие в `palata_status_events`.
8. Создаётся test email event в `palata_email_events`.

Проверить:

* реализована ли кнопка;
* правильно ли блокируются остальные эксперты;
* корректно ли закрываются action items;
* пишутся ли события.

---

## 5.2. Эксперт нажал «Могу начать с даты»

Ожидаемое поведение:

1. Текущий match = `can_start_from`.
2. Сохраняется дата начала.
3. Action item эксперта закрывается.
4. Заказчику создаётся action item:
   `expert_can_start_from`.
5. Создаётся событие в `palata_status_events`.
6. Создаётся test email event в `palata_email_events`.

Проверить:

* есть ли выбор даты;
* где сохраняется дата;
* создаётся ли action item заказчику;
* закрывается ли action item эксперта.

---

## 5.3. Эксперт нажал «Отказаться»

1. Текущий match = `declined`.
2. Сохраняется причина отказа.
3. Сохраняется комментарий.
4. Action item эксперта закрывается.

# СЦЕНАРИЙ 6. Заказчик отвечает на предложенную дату

Если эксперт предложил дату, заказчик получает action item:

`expert_can_start_from`

Заказчик может выбрать:

1. «Согласовать дату»
2. «Выбрать другого эксперта»

---

## 6.1. Заказчик согласовал дату

Важная логика:
после согласования даты заказчиком заказ НЕ должен сразу переходить в `in_work`.

Причина:
заказчик мог долго обрабатывать ответ, и эксперт мог стать недоступен.

Ожидаемое поведение:

1. Action item заказчика `expert_can_start_from` закрывается.
2. Эксперту создаётся новый action item:
   `you_are_approved_for_work`.
3. Заказ пока НЕ переводится в `in_work`.
4. Остальные эксперты пока НЕ закрываются.
5. Создаётся событие:
   `customer_approved_start_date`.
6. Создаётся test email event.

Проверить:

* не переводится ли заказ в работу слишком рано;
* создаётся ли action item эксперту;
* не закрываются ли остальные эксперты преждевременно.

---

## 6.2. Заказчик выбрал другого эксперта

Ожидаемое поведение:

1. Текущий match получает статус:
   `customer_declined_start_date`.
2. Заказ остаётся в `expert_selection`.
3. Заказчик может выбрать другого эксперта из ранее подобранных.
4. Создаётся событие.
5. Создаётся test email event.

Проверить:

* есть ли такой сценарий;
* не ломается ли список экспертов;
* можно ли выбрать другого.

---

# СЦЕНАРИЙ 7. Эксперт подтверждает финальное назначение

После того как заказчик согласовал дату эксперта, эксперт получает action item:

`you_are_approved_for_work`

Название:
`Вы назначены на заказ`

Эксперт видит две кнопки:

1. «ОК, беру в работу»
2. «Отказаться»

---

## 7.1. Эксперт нажал «ОК, беру в работу»

Тогда запускается стандартный сценарий взятия в работу:

1. `palata_requests.status = in_work`.
2. Текущий match эксперта = `accepted_work`.
3. Остальные matches по этому request_id = `closed_by_other_expert`.
4. Все открытые action items других экспертов по заказу закрываются как `cancelled`.
5. Action item эксперта `you_are_approved_for_work` закрывается.
6. Заказчику создаётся action item / событие:
   `expert_started_work`.
7. Создаётся запись в `palata_status_events`.
8. Создаётся test email event.

Проверить:

* реализован ли action type `you_are_approved_for_work`;
* есть ли кнопка подтверждения;
* только после неё ли заказ переходит в работу.

---

## 7.2. Эксперт нажал «Отказаться»

Тогда запускается стандартный сценарий отказа:

1. Текущий match эксперта = `declined`.
2. Сохраняется причина отказа.
3. Сохраняется комментарий.
4. Action item эксперта закрывается.
5. Заказчику создаётся action item:
   `expert_declined`.
6. Создаётся событие.
7. Создаётся test email event.

Проверить:

* работает ли отказ после финального назначения;
* возвращается ли заказчик к выбору другого эксперта.

---

# СЦЕНАРИЙ 8. Все эксперты отказались

Если все подобранные эксперты по заказу отказались:

Ожидаемое поведение:

1. Заказ возвращается в `matching`.
2. `matching_round` увеличивается на 1.
3. Запускается новый подбор.
4. Эксперты, которые уже отказались, исключаются.
5. Если новые эксперты найдены:

   * заказчику создаётся новый action item `experts_matched`.
6. Если новые эксперты не найдены:

   * админу создаётся action item `manual_matching_required`.

Проверить:

* реализован ли повторный подбор;
* не попадают ли отказавшиеся эксперты повторно;
* увеличивается ли matching_round.

---

# СЦЕНАРИЙ 9. Завершение заказа экспертом

Только эксперт может перевести заказ в `completed`.

Заказчик НЕ должен иметь кнопку «Перевести в выполнено».

При завершении эксперт:

1. Нажимает «Завершить заказ».
2. Ставит оценку заказчику от 1 до 5.
3. Оставляет комментарий.
4. Заказ получает статус `completed`.
5. Match эксперта получает статус `completed`.
6. Создаётся запись в `palata_customer_ratings`.
7. Заказчику создаётся action item:
   `expert_completed_order`.
8. Создаётся событие.
9. Создаётся test email event.

Проверить:

* нет ли кнопки завершения у заказчика;
* есть ли оценка заказчика экспертом;
* создаётся ли action item заказчику.

---

# СЦЕНАРИЙ 10. Заказчик оценивает эксперта

После завершения заказа экспертом заказчик видит задачу:

`Оценить эксперта`

Ожидаемое поведение:

1. Задача отображается во вкладке:
   `Оценить экспертов`.
2. Заказчик ставит оценку эксперту от 1 до 5.
3. Заказчик оставляет комментарий.
4. Создаётся запись в `palata_expert_ratings`.
5. Action item закрывается.
6. Создаётся событие.
7. Рейтинг эксперта пересчитывается.

Проверить:

* есть ли вкладка «Оценить экспертов»;
* отображаются ли нужные заказы;
* создаётся ли оценка;
* закрывается ли action item.

# СЦЕНАРИЙ 11. Badge-счётчики

Badge на вкладках должен считаться так:

## Для «Требуют действия»

Условия:

* `assigned_to_user_id = current user`;
* `status = open`;
* `is_resolved = false`.

Дополнительно:

* если нужно считать только новые — учитывать `is_read = false`.

## Для «Оценить экспертов»

Условия:

* заказ completed;
* есть action item `expert_completed_order`;
* ещё нет оценки в `palata_expert_ratings`.

## Для «Оценить заказчиков»

Условия:

* заказ completed;
* текущий эксперт ещё не поставил оценку заказчику.

Проверить:

* как сейчас считаются badge;
* нет ли дублей;
* не считаются ли закрытые action items.
