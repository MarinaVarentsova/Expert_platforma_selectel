Нужно переделать справочник направлений экспертиз на единый источник данных в Supabase.

Сейчас по аудиту:

* в БД нет справочной таблицы направлений;
* `palata_requests.expertise_type` хранит свободный текст;
* `palata_expert_profiles.specializations` хранит массив slug;
* в коде есть 6 разных несинхронизированных справочников.

Это нужно исправить.

# Целевая архитектура

## 1. Создать таблицу справочника

Создать таблицу:

`palata_expertise_directions`

Поля:

* id uuid primary key
* name text not null unique
* slug text not null unique
* sort_order integer
* is_active boolean default true
* created_at timestamptz default now()
* updated_at timestamptz default now()

Источник истины — только эта таблица.

# 2. Заполнить таблицу только этими направлениями

В алфавитном порядке:

* Автотехническая экспертиза
* Баллистическая экспертиза
* Биологическая экспертиза
* Взрывотехническая экспертиза
* Другие экспертизы
* Землеустроительная экспертиза
* Исследование объектов судебной экспертизы с применением инструментально-лабораторных методов
* Компьютерно-техническая экспертиза
* Криминалистическая экспертиза видео- и звукозаписей
* Лингвистическая экспертиза
* Молекулярно-генетическая экспертиза объектов
* Оценочная экспертиза
* Патентная экспертиза
* Пожарно-техническая экспертиза
* Политологическая экспертиза
* Портретная экспертиза
* Почвоведческая экспертиза
* Почерковедческая экспертиза
* Психологическая экспертиза
* Судебно-медицинская экспертиза
* Строительно-техническая экспертиза
* Товароведческая экспертиза
* Трасологическая экспертиза
* Финансово-экономическая экспертиза
* Экологическая экспертиза
* Экспертиза веществ, материалов, изделий
* Экспертиза маркировочных обозначений
* Экспертиза объектов дикой флоры и фауны
* Экспертиза охраны труда и техники безопасности
* Экспертиза холодного оружия
* Электротехническая экспертиза

Другие направления не использовать.

# 3. Заказы должны хранить ID направления

В `palata_requests` добавить поле:

`expertise_direction_id uuid references palata_expertise_directions(id)`

Дальше создание заказа должно сохранять именно `expertise_direction_id`, а не свободный текст.

Старое поле `expertise_type` пока не удалять, но больше не использовать как источник истины.

# 4. Эксперт может выбрать несколько направлений

Создать связующую таблицу:

`palata_expert_directions`

Поля:

* id uuid primary key
* expert_id uuid not null
* expertise_direction_id uuid not null references palata_expertise_directions(id)
* created_at timestamptz default now()

Ограничение:

unique(expert_id, expertise_direction_id)

Эксперт может выбрать несколько направлений.

# 5. Формы должны использовать только таблицу

Формы должны загружать направления из:

`palata_expertise_directions`

Использовать эту таблицу в:

* регистрации эксперта;
* редактировании профиля эксперта;
* создании заказа;
* редактировании заказа;
* фильтрах;
* метриках;
* matching engine.

Удалить / заменить локальные справочники:

* `EXPERTISE_TYPES`
* `SPEC_OPTIONS`
* `EXPERTISE_ALIASES`
* `EXPERTISE_LABEL`
* `SPEC_LABEL`
* `SPEC_L`

Их больше не должно быть как источника истины.

# 6. Matching engine

Подбор экспертов должен сравнивать ID направления.

Логика:

`palata_requests.expertise_direction_id`
сравнивается с
`palata_expert_directions.expertise_direction_id`

Если совпадает — эксперт подходит по направлению.

Больше не использовать fuzzy matching, slug aliases и ручные словари.

# 7. Миграция старых данных

Сделать migration, которая:

1. Создаёт `palata_expertise_directions`.
2. Заполняет её эталонным списком.
3. Добавляет `expertise_direction_id` в `palata_requests`.
4. Создаёт `palata_expert_directions`.
5. Переносит старые данные:

   * старые русские названия из `palata_requests.expertise_type` сопоставить с новым справочником;
   * старые slug из `palata_expert_profiles.specializations` сопоставить с новым справочником.
6. Не удаляет старые поля до отдельного подтверждения.

# 8. Важно

Не ломать существующие заказы и профили.

Если старое значение не удаётся сопоставить — ставить направление:

`Другие экспертизы`
