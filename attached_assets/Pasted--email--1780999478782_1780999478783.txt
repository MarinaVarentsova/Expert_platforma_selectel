КРИТИЧЕСКИЙ БАГ. Нужно срочно исправить регистрацию и подтверждение email.

Кредиты и сроки ограничены, поэтому без долгих аудитов и без лишних экспериментов.
SMTP/Resend НЕ трогать — он работает.
Supabase SMTP настроен, письма уходят и доставляются.

Сейчас факты:

1. Пользователь нажимает "Зарегистрироваться".
2. В Supabase Auth пользователь создаётся.
3. В palata_customer_profiles запись создаётся.
4. Письмо через Resend отправляется.
5. Но в интерфейсе показывается ошибка:
   "Произошла ошибка. Попробуйте ещё раз."

Это неверно.

После успешного signUp при включённом Confirm Email пользователь должен увидеть сообщение:
"Спасибо за регистрацию. На почту <email> отправлено письмо для подтверждения регистрации. Перейдите по ссылке из письма, чтобы завершить регистрацию."

НЕ показывать ошибку, если Auth-пользователь и профиль созданы, а письмо отправлено.

Нужно найти место в Register.tsx / auth flow, где после успешного signUp падает код.

Исправить:

- если supabase.auth.signUp() успешен;
- если data.user есть;
- если data.session === null;

то это НЕ ошибка, а нормальный сценарий email confirmation.

Показать success-screen/message и остановить дальнейший переход в ЛК.

ПРОБЛЕМА 2. Дубли пользователей по одному email

Сейчас один и тот же email может зарегистрироваться несколько раз и создаёт несколько UID в Supabase Auth.

Нужно исправить.

Перед signUp проверить:

- есть ли уже пользователь/профиль с таким email в public.palata_users;
- если есть, не вызывать signUp повторно;
- показать:
  "Пользователь с данной почтой уже зарегистрирован. Войдите в систему или восстановите пароль."

Важно:
не создавать повторные записи в:
- auth.users
- palata_users
- palata_customer_profiles
- palata_expert_profiles

Если Supabase из-за настроек позволяет повторный unconfirmed signup, всё равно на фронте и/или через БД должна быть защита от дублей.

Проверить наличие unique constraint на public.palata_users.email.
Если его нет — дать SQL, но без самовольного применения миграции:
ALTER TABLE public.palata_users ADD CONSTRAINT palata_users_email_unique UNIQUE (email);

ПРОБЛЕМА 3. Ссылка из письма ведёт на /auth/callback и отдаёт 404

Сейчас письмо содержит ссылку на:
https://expertplatforma.vercel.app/auth/callback...

Но маршрута /auth/callback в приложении нет или он не работает.
Vercel отдаёт 404.

Нужно создать рабочий маршрут /auth/callback.

Логика /auth/callback:

1. Прочитать параметры URL.
2. Если Supabase вернул code:
   выполнить supabase.auth.exchangeCodeForSession(code)

3. Если в URL пришла ошибка:
   error=access_denied
   error_code=otp_expired
   показать понятное сообщение:
   "Ссылка подтверждения недействительна или истекла. Зарегистрируйтесь повторно или запросите новое письмо."

4. После успешной сессии:
   получить текущего пользователя:
   supabase.auth.getUser()

5. Найти роль пользователя в public.palata_users по id = user.id.

6. Перенаправить:
   customer → /customer
   expert → /expert
   admin → /admin

Если в проекте используются другие реальные маршруты ЛК — использовать существующие маршруты проекта.

ПРОБЛЕМА 4. Подтверждение email должно отражаться в Supabase Auth

После клика по кнопке "Подтвердить почту" Supabase должен проставить:
auth.users.email_confirmed_at IS NOT NULL

Нужно проверить, что после обработки /auth/callback:
- email_confirmed_at заполнен;
- пользователь получает session;
- пользователь попадает в ЛК.

ПРОБЛЕМА 5. Очистить битые refresh-token ошибки

После удаления пользователей из Supabase Auth браузер может хранить старую сессию.
Если возникает:
Invalid Refresh Token: Refresh Token Not Found

нужно:
- выполнить supabase.auth.signOut()
- очистить локальную сессию
- не блокировать регистрацию
- не показывать пользователю общую ошибку регистрации из-за старого refresh token.
