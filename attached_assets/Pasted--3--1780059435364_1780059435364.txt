По результатам аудита нужно исправить только 3 расхождения с эталонным сценарием.

Пока больше ничего не менять.

# 1. Контакты должны открываться сразу после выбора эксперта

Сейчас контакты открываются только после того, как эксперт перевёл заказ в работу.

Это не соответствует целевому сценарию.

После выбора эксперта заказчиком нужно:

1. создать или обновить запись в `palata_request_contacts`;
2. сохранить:

   * request_id;
   * expert_id;
   * customer_id;
   * contact_opened_at = now();
3. сразу открыть заказчику контакты эксперта;
4. сразу открыть эксперту контакты заказчика.

Причина:

После выбора эксперта стороны уже могут созвониться, обсудить детали заказа, документы, сроки и организационные вопросы ещё до старта работ.

Проверить:

* где сейчас проверяется право просмотра контактов;
* какие условия нужно изменить;
* какие данные уже есть в базе.

---

# 2. Не переводить заказ в работу после согласования даты

Сейчас реализован сценарий:

Эксперт:
`Могу начать с даты`

↓

Заказчик:
`Согласовать дату`

↓

Заказ сразу получает статус:

`in_work`

↓

Остальные эксперты получают:

`closed_by_other_expert`

Это неверно.

После согласования даты заказчиком НЕ нужно:

* переводить заказ в `in_work`;
* переводить выбранного эксперта в `accepted_work`;
* закрывать остальных экспертов;
* отменять action items остальных экспертов.

После согласования даты нужно только:

1. закрыть action item заказчика;
2. создать новый action item эксперту;
3. создать событие в `palata_status_events`;
4. создать test email event.

Отдельно проверить:

* `handleApprove`;
* все места, где выставляется `in_work`;
* все места, где выполняется `closed_by_other_expert`;
* все места, где отменяются action items других экспертов.

# 3. Добавить финальное подтверждение эксперта

Добавить новый action type:

`you_are_approved_for_work`

Название:

`Вы назначены на заказ`

Описание:

`Заказчик подтвердил выбор вас как исполнителя. Подтвердите готовность взять заказ в работу.`

После согласования даты заказчиком:

1. action item заказчика закрывается;
2. создаётся action item эксперту:
   `you_are_approved_for_work`;
3. заказ остаётся в текущем статусе;
4. остальные эксперты остаются активными;
5. никаких блокировок не происходит.

---

# 4. Новый сценарий для эксперта

Когда эксперт получает:

`you_are_approved_for_work`

он должен видеть две кнопки:

### ОК, беру в работу

Только здесь выполняется:

* `palata_requests.status = in_work`;
* текущий match = `accepted_work`;
* остальные matches по заказу =
  `closed_by_other_expert`;
* action items остальных экспертов =
  `cancelled`;
* action item эксперта =
  `resolved`;
* создаётся action item заказчику:
  `expert_started_work`;
* создаётся событие;
* создаётся test email event.

### Отказаться

Стандартный сценарий отказа:

* match = `declined`;
* сохраняется причина отказа;
* сохраняется комментарий;
* action item эксперта закрывается;
* заказчику создаётся:
  `expert_declined`;
* создаётся событие;
* создаётся test email event.

# Важно

Зафиксировать отдельно:

После согласования даты заказчиком остальные эксперты НЕ должны закрываться.

Не выполнять:

* `closed_by_other_expert`;
* отмену action items;
* блокировку выбора.

Остальные эксперты закрываются только после того, как выбранный эксперт нажал:

`ОК, беру в работу`

Именно этот момент считается окончательным подтверждением назначения.
