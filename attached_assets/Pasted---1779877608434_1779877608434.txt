Нужно добавить новый функционал: «Требуют действия» — обмен статусами заказа между заказчиком и экспертом.

Важно:
канбан-доски НЕ удалять.
Канбан остаётся как общий обзор заказов по статусам.
Новый функционал «Требуют действия» — это дополнительный workflow-inbox, где пользователь видит только те заказы, по которым сейчас требуется его решение.

# 1. Общая логика

В системе есть две разные зоны:

## Канбан

Показывает общее состояние заказов:

* новые;
* подбор;
* выбор эксперта;
* в работе;
* выполнено;
* неактуальные.

## Требуют действия

Показывает только те заказы, где текущий пользователь должен что-то сделать:

* выбрать эксперта;
* ответить по заказу;
* согласовать дату;
* отказаться;
* взять в работу;
* оценить другую сторону.

В навигации вкладка должна показывать бейдж с количеством непросмотренных / нерешённых действий:

Пример:
`Требуют действия 2`

Количество = action items, где:

* assigned_to_user_id = текущий пользователь;
* is_resolved = false;
* is_read = false или viewed_at is null.

# 3. Вкладки в ЛК заказчика

В `/customer` добавить вкладки:

1. Мои заказы
2. Требуют действия
3. Оценить экспертов

Вкладка «Мои заказы»:

* оставить текущий канбан.

Вкладка «Требуют действия»:
показывать action items заказчика.

Туда должны попадать:

## 1. Система подобрала экспертов

Action type:
`experts_matched`

Показывать:

* заказ;
* список подобранных экспертов;
* кнопку «Посмотреть экспертов».

Заказчик открывает список экспертов:

* видит карточки экспертов;
* может открыть мини-профиль эксперта;
* может выбрать эксперта.

Карточка эксперта должна показывать:

* ФИО;
* рейтинг;
* регионы;
* направления;
* готовность к командировкам;
* регистрация в Палата судебных экспертов РФ;
* номер регистрации;
* регистрация в Центр судэксперт;
* номер регистрации;
* количество выполненных заказов;
* краткое описание опыта.

Действие заказчика:
`Выбрать эксперта`

После выбора эксперта:

* обновить `palata_request_matches`;
* создать / обновить `palata_request_contacts`;
* создать action item эксперту:
  `customer_selected_you`;
* action item заказчика `experts_matched` пометить resolved;
* записать событие в `palata_status_events`;
* записать email-event в `palata_email_events` в test mode.

## 2. Эксперт отказался

Action type:
`expert_declined`

Показывать:

* какой эксперт отказался;
* причина отказа;
* комментарий;
* кнопку «Выбрать другого эксперта».

После выбора другого эксперта:

* создать action item новому эксперту;
* старый отказ оставить в истории;
* событие записать в `palata_status_events`.

## 3. Эксперт может начать с даты

Action type:
`expert_can_start_from`

Показывать:

* эксперт;
* предложенная дата начала;
* комментарий эксперта.

Действия заказчика:

* «Согласовать дату»
* «Выбрать другого эксперта»

Если заказчик нажал «Согласовать дату»:

* статус заказа = `in_work`;
* статус выбранного эксперта по заказу = `accepted_work`;
* остальные эксперты по этому заказу = `closed_by_other_expert`;
* action item заказчика = resolved;
* создать action item эксперту:
  `customer_approved_start_date`;
* записать событие;
* записать email-event.

Если заказчик нажал «Выбрать другого эксперта»:

* текущий expert match получает статус `customer_declined_date`;
* заказчик возвращается к списку подобранных экспертов;
* action item остаётся open или создаётся новый `choose_another_expert`.

## 4. Заказ завершён экспертом

Action type:
`expert_completed_order`

Показывать:

* заказ;
* эксперт;
* дату завершения;
* кнопку «Оценить эксперта».

После оценки:

* записать оценку в `palata_expert_ratings`;
* action item закрыть;
* событие записать в `palata_status_events`.

# 5. Логика выбора и блокировки остальных экспертов

Когда заказчик выбрал одного эксперта:

* выбранный эксперт получает action item;
* остальные эксперты пока не получают задачу, если заказчик их не выбрал.

Если эксперт взял заказ в работу:

* `palata_requests.status = in_work`;
* выбранный expert match = `accepted_work`;
* остальные matches по этому request_id:
  `closed_by_other_expert`;
* все открытые action items по этому заказу для других экспертов закрываются со статусом `cancelled`.

Если эксперт отказался:

* отказавшийся expert match = `declined`;
* заказчик получает action item `expert_declined`;
* заказчик может выбрать другого из уже подобранных экспертов;
* если все подобранные эксперты отказались:

  * заказ возвращается в `matching`;
  * увеличивается `matching_round`;
  * новый подбор исключает экспертов, которые уже отказались.

* если действие выполнено — is_resolved = true.
