Берём следующий бизнес-сценарий:
«Автоматический matching — подбор экспертов».

Файл:
artifacts/palata/src/lib/matching.ts

Критически важно:
только переключение источника данных.
Бизнес-логику matching сохранить один в один.

- исключения;
- matching_round;
- статусы;

- status events;
- тексты;

Сначала провести точный аудит runMatching():

1. Какие данные получает функция.
2. Какие таблицы читает.
3. Какие фильтры и исключения применяет.
4. Как вычисляется matching_round.
5. Как исключаются ранее отказавшиеся эксперты.
6. Как определяется список кандидатов.
7. Как выбираются максимум 5 экспертов.
8. Какие строки вставляются в palata_request_matches.
9. Как меняется palata_requests.status.
10. Какие status events создаются.
11. Какие action items создаются.
12. Какие действия выполняются при отсутствии экспертов.
13. Какие данные нужны frontend после завершения.

После аудита реализовать backend endpoint:

POST /api/palata/requests/:requestId/matching/run

Bearer token обязателен.

Доступ:
- admin;
- customer-владелец заявки;
- либо внутренний серверный вызов из уже существующего сценария отказа, если он реализован через тот же frontend token.

Не принимать customer_id, expert_id или matching_round из body без проверки.
requestId брать из URL.

Внутри PostgreSQL-транзакции через PALATA_DATABASE_URL выполнить только текущую write-логику matching:

BEGIN;

1. Загрузить заявку:
   - id;
   - status;
   - customer_id;
   - expertise_direction_id;
   - region_id;
   - requires_travel;
   - matching_round.

2. Вычислить следующий matching_round точно так же, как сейчас.

3. Получить ранее созданные matches и исключить экспертов по тем же правилам.

4. Получить кандидатов тем же способом, что сейчас:
   - действующий сертификат;
   - нужное направление;
   - accepts_requests=true;
   - регион при requires_travel=true;
   - остальные существующие фильтры.

Важно:
уже мигрированные GET endpoints можно использовать сервером только если это не меняет логику.
Предпочтительно читать напрямую PostgreSQL из:
- palata_expert_certificates;
- palata_expert_profiles;
- palata_expert_regions;
- palata_users.

5. Сохранить текущий порядок сортировки и выбора кандидатов.

6. Если кандидаты найдены:
   - INSERT до 5 строк в palata_request_matches
     со статусом 'proposed'
     и текущим matching_round;
   - UPDATE palata_requests:
     status='expert_selection',
     matching_round=nextRound;

8. COMMIT.
