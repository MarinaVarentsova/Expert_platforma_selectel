# DEF-01 CRITICAL

## Кнопка «Выбрать эксперта» не должна зависеть от открытого action item

Сейчас критический баг:

Кнопка `Выбрать эксперта` появляется только если есть открытый action item:

`experts_matched`

Это неверно.

Если action item уже закрыт, потерян или не был создан, но заказ всё ещё находится в статусе:

`expert_selection`

заказчик всё равно должен иметь возможность выбрать эксперта.

## Правильная логика

Кнопка `Выбрать эксперта` должна отображаться по бизнес-состоянию заказа, а не по наличию action item.

Показывать кнопку, если одновременно выполнены условия:

1. текущий пользователь = заказчик этого заказа;
2. `request.status = expert_selection`;
3. есть записи в `palata_request_matches` по этому заказу;
4. конкретный match эксперта ещё доступен для выбора.

## Match доступен для выбора, если его status НЕ входит в:

* `declined`;
* `closed_by_other_expert`;
* `withdrawn`;
* `accepted_work`;
* `completed`;
* `customer_declined_start_date`.

Если статус match:

* `accepted`;
* `selected_by_customer`;
* `can_start_from`;
* `pending`;
* `matched`;
* null / пустой,

то эксперт должен быть доступен для выбора, если заказ ещё в `expert_selection`.

## Важно

Action item `experts_matched` — это только уведомление / вход в сценарий.

Он НЕ должен быть единственным условием для отображения кнопки.

Нельзя больше писать логику вида:

`show button only if expertsMatchedItem exists`

Нужно заменить её на:

`show button if request is in expert_selection and match is selectable`.

## Что проверить

Проверить файл:

`RequestDetail.tsx`

Особенно место, где сейчас кнопка зависит от:

`expertsMatchedItem`

Нужно убрать жёсткую зависимость от open action item.

## Критерий готовности DEF-01

1. Заказчик открывает заказ в статусе `expert_selection`.
2. У заказа есть доступные matches.
3. Даже если action item `experts_matched` уже resolved или отсутствует, заказчик видит кнопку `Выбрать эксперта`.
4. После выбора эксперта текущий workflow работает как раньше:

   * match обновляется;
   * contacts создаются/обновляются;
   * action item эксперту создаётся;
   * status event создаётся;
   * email event создаётся.

# DEF-02 HIGH

## Админ должен запускать подбор для заказа в статусе matching

Сейчас админ не видит кнопку:

`Запустить подбор экспертов`

для заказа в статусе:

`matching`

Это неверно.

Админ должен иметь возможность вручную перезапустить подбор, если заказ завис в подборе или эксперты не найдены.

## Правильная логика

Показывать кнопку:

`Запустить подбор экспертов`

если:

1. `role = admin`;
2. `request.status IN ('new', 'pending', 'matching')`;
3. request.status НЕ равен:

   * `completed`;
   * `not_actual`;
   * `cancelled`;
   * `archived`.

## При нажатии

Вызвать существующий механизм:

`runMatching()`

НЕ создавать новый параллельный matching engine.

## Обязательные правила

При запуске matching:

1. Не создавать дубли matches.
2. Не добавлять повторно экспертов, которые уже есть в matches по этому заказу, если их статус активный.
3. Не возвращать в подбор экспертов, которые уже отказались:

   * `declined`;
   * `customer_declined_start_date`;
   * `withdrawn`.
4. Учитывать текущий matching_round.
5. Если найдены новые эксперты:

   * создать/обновить `palata_request_matches`;
   * перевести заказ в `expert_selection`;
   * создать или обновить action item заказчику `experts_matched`;
   * создать событие в `palata_status_events`.
6. Если эксперты не найдены:

   * оставить / вернуть заказ в `matching`;
   * создать action item админу:
     `manual_matching_required`;
   * создать событие:
     `no_experts_found`.

## Что проверить

Проверить файл:

`RequestDetail.tsx`

Особенно место, где сейчас кнопка `Запустить подбор` показывается только customer или не показывается admin.

## Критерий готовности DEF-02

1. Админ открывает заказ в статусе `matching`.
2. Видит кнопку `Запустить подбор экспертов`.
3. Нажимает кнопку.
4. Подбор запускается через существующий `runMatching()`.
5. Если эксперты найдены — заказ переходит в `expert_selection`.
6. Если эксперты не найдены — заказ остаётся в `matching`, админу создаётся `manual_matching_required`.
7. Дубли matches не создаются.
