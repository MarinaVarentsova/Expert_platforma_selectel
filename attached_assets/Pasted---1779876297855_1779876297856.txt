Нужно зафиксировать единые формулы расчёта метрик.

Сейчас цифры выглядят несогласованно:
например:

* активных заказчиков больше, чем всего заказчиков;
* активных экспертов = 0 при наличии заказа в работе;
* проценты считаются по разным основаниям.

Нужно привести все показатели к единой логике.

# ЕДИНЫЕ ФОРМУЛЫ МЕТРИК

## 1. Верхняя строка KPI

### Всего заявок

Источник:
`palata_requests`

Формула:

```sql
COUNT(*)
```

Учитываются:

* все заявки кроме soft deleted (если soft delete есть)

---

### Выполнено заявок

Источник:
`palata_requests`

Формула:

```sql
COUNT(*) WHERE status = 'completed'
```

---

### % выполненных

Формула:

```text
completed_requests / total_requests * 100
```

Пример:

```text
4 / 16 = 25%
```

---

### Среднее время выполнения (дни)

Источник:
`palata_requests`

Формула:

```text
AVG(completed_at - created_at)
```

Важно:

* считать только completed заявки;
* брать дату завершения:

  * либо `completed_at`,
  * либо последнее событие completed из `palata_status_events`.

НЕ использовать:

```text
updated_at
```

иначе среднее время будет искажаться.

---

# 2. Правая панель

## Средний рейтинг эксперта

Источник:
`palata_expert_ratings`

Формула:

```sql
AVG(score)
```

Считать:

* только реальные оценки;
* без NULL;
* без тестовых пустых строк.

---

## Средний рейтинг заказчика

Источник:
`palata_customer_ratings`

Формула:

```sql
AVG(score)
```

---

## Всего заказчиков

Источник:
`palata_customer_profiles`

Формула:

```sql
COUNT(*)
```

---

## Активные заказчики

Определение:
заказчик считается активным, если у него есть хотя бы одна заявка НЕ в статусе:

* not_actual
* cancelled
* archived

Формула:

```sql
COUNT(DISTINCT customer_id)
WHERE request.status NOT IN (...)
```

---

## % активных заказчиков

Формула:

```text
active_customers / total_customers * 100
```

ВАЖНО:
активных НЕ МОЖЕТ быть больше, чем всего.

Если сейчас:

```text
7 активных из 6
```

значит:

* неправильный join;
* дубли customer_id;
* count без distinct.

Нужно использовать:

```sql
COUNT(DISTINCT customer_id)
```

---

## Всего экспертов

Источник:
`palata_expert_profiles`

Формула:

```sql
COUNT(*)
```

---

## Активные эксперты

Определение:
эксперт считается активным, если у него есть хотя бы один заказ:

```text
status = in_work
```

Источник:
`palata_request_matches`

Формула:

```sql
COUNT(DISTINCT expert_id)
WHERE match_status = 'accepted_work'
```

или:

```sql
WHERE request.status = 'in_work'
```

ВАЖНО:
если есть 1 заказ в работе,
активных экспертов не может быть 0.

---

## % активных экспертов

Формула:

```text
active_experts / total_experts * 100
```

---

## Реестр Палаты СЭ

Источник:
`palata_expert_profiles`

Формула:

```sql
COUNT(*)
WHERE palata_registry_verified = true
```

---

## Центр судэксперт

Источник:
`palata_expert_profiles`

Формула:

```sql
COUNT(*)
WHERE centrsudexpert_verified = true
```

---

# 3. Основная дорожка заявок

Основание процентов:

```text
total_requests
```

То есть:
каждый % считается от:

```text
всего заявок
```

---

## Всего

```sql
COUNT(*)
```

---

## Новые

```sql
COUNT(*) WHERE status = 'new'
```

---

## Подбор эксперта

```sql
COUNT(*) WHERE status = 'matching'
```

---

## Выбор эксперта

```sql
COUNT(*) WHERE status = 'expert_selection'
```

---

## В работе

```sql
COUNT(*) WHERE status = 'in_work'
```

---

## Выполнено

```sql
COUNT(*) WHERE status = 'completed'
```

---

## Неактуальные

```sql
COUNT(*) WHERE status IN (
  'not_actual',
  'cancelled',
  'archived'
)
```

---

# 4. Проблемная дорожка

## Не нашли эксперта

Определение:

* заказ находится в matching;
* и нет ни одного активного match;
  ИЛИ
* есть event `no_experts_found`.

---

## Отказ эксперта

Определение:
есть хотя бы один:

```text
match_status = declined
```

Источник:
`palata_request_matches`

Формула:

```sql
COUNT(DISTINCT request_id)
WHERE match_status = 'declined'
```

---

## Неактуальные

Берём те же статусы,
что и в основной дорожке.

---

# 5. Распределения

## Заказы → по регионам

Источник:
`palata_requests.region`

Формула:

```sql
COUNT(*) GROUP BY region
```

---

## Заказы → по направлениям

Источник:
`palata_requests.expertise_type`
или фактическое поле направления.

---

## Эксперты → по регионам

Источник:
`palata_expert_profiles.regions`

Если массив:
использовать unnest.

---

## Эксперты → по направлениям

Источник:
`palata_expert_profiles.specializations`

Если массив:
использовать unnest.

---

# ВАЖНО

Все метрики должны считаться:

* из одного metrics service;
* с единым mapping статусов;
* с единым source of truth.

Нельзя:

* в одном блоке считать completed по requests;
* а в другом по status_events;
* или использовать разные фильтры.

После исправления:

1. показать SQL/TS формулы;
2. показать mapping статусов;
3. показать какие DISTINCT используются;
4. объяснить почему цифры теперь не расходятся.
